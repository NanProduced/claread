# TMP Reader 用户学习资产重审

## 状态

- 类型：TMP
- 日期：2026-05-20
- 范围：Web Reader / 小程序 Reader 的用户高亮、笔记、收藏，以及 `/library/assets` “摘录与批注”模块
- 目的：冻结本轮评估结论，为后续产品重定义提供输入

## 背景

本轮 Reader UI 改造过程中，围绕用户高亮、笔记、收藏的冲突处理、重叠规则、回显方式和“学习资产”产品定位进行了多轮评估。结论是：问题不只是前端样式或交互细节，而是当前需求定义本身过早、过宽，导致前后端对象模型和页面呈现一起复杂化。

## 对微信读书的吸收

本轮最值得吸收的不是微信读书的具体视觉样式，而是它处理冲突的产品约束：

1. 同一选中段只存在一个底层批注对象。
2. 已有批注再次命中时，优先进入“编辑已有对象”，而不是默认新建。
3. 子串 / 部分重叠的复杂度在规则层被消解，而不是留给渲染层兜底。

这一点与 Claread 当前“允许并存、允许重叠、再靠 stacked 渲染兜底”的实现思路不同。后者在技术上可行，但在产品层没有闭环，会持续放大：

- `sentence / text_range / multi_text`
- `annotation / favorite`
- highlight / note
- overlap / partial overlap / multi overlap

这些维度之间的冲突成本。

## 当前问题判断

### 1. 用户学习资产对象模型过宽

当前实现同时开放：

- 句子级用户批注
- 单句内 `text_range`
- 跨句/跨段 `multi_text`
- 文本收藏
- 笔记与高亮并行
- `/library/assets` 聚合展示

这使得 Reader 主场之外又生成了一条“资产副场”，但这条副场暂时没有提供强于 Reader 本身的学习价值。

### 2. “学习资产”产品定位不成立

`/library/assets` 当前更接近：

- 摘录索引
- 批注归档
- 文章内用户痕迹的二次列表

而不是成熟的“学习资产中心”。

它没有形成：

- 跨文章知识组织
- 系统化复习单元
- 弱项聚合
- 练习/回顾工作流

但已经开始给 Reader 主场施加复杂的锚点、冲突和回跳成本。

### 3. 模块价值弱于解析页本身

当前一个解析页已经完整呈现：

- 原文
- 译文
- 机器解析
- 用户高亮
- 用户笔记

在这种前提下，再做一个“把这些对象重新按文章列一次”的页面，用户收益很有限，甚至比直接回看解析页更杂乱。

## 当前冻结结论

### A. Reader 主场优先

继续强化解析页本身：

- 文字底板
- 机器批注
- 用户高亮 / 笔记
- AI grounded chat

而不是继续扩展“学习资产副场”。

### B. 文本收藏进入待移除评估

文章收藏仍有价值。  
文本收藏当前价值不足，且会额外引入平行对象和冲突规则，进入待移除评估。

### C. `/library/assets` 进入产品重审

该页面当前不再视为稳定长期方向。  
后续要么降级为内部/过渡索引，要么在真正的学习资产产品模型明确后重做。

### D. `multi_text` 用户学习资产进入待重审

`multi_text` 作为 Ask / 临时操作 / 分享锚点仍有技术价值；  
但它是否继续作为用户高亮/笔记/收藏的长期 authoring 目标，不再默认成立。

## 后续建议

1. 先暂停“学习资产中心”方向，不继续打磨 `/library/assets`。
2. Reader 内保留用户高亮、用户笔记、文章收藏，继续围绕阅读主场收口。
3. 等真正的复习、回顾、知识重组需求明确后，再重新定义“学习资产”。
4. 在此之前，相关稳定文档统一降级表述，不把当前实现误写成既定产品方向。
