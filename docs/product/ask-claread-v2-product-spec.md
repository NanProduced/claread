# Ask Claread V2 Product Spec

## 文档状态

- 状态：active redesign target
- 日期：2026-05-18
- 适用范围：Claread Web Reader 内 Ask Claread 重构
- 事实基线：本文以当前代码实现与可验证产品调研为依据，不以旧文档交互口径为准
- 文档关系：`docs/product/ask-claread-v1.md` 仅保留为第一版对照；开发期进度与决策文档统一维护在 `docs/tmp/ask-claread-rebuild/`

## 背景

当前 Ask Claread 已经具备可用的 LLM 调用、文章绑定线程、SSE 流式输出、引用、结构化卡片与保存确认动作，但整体体验仍然偏“AI 控制台”，不符合 Claread 作为阅读与解析产品的核心场景。

当前主要问题不只是 UI/UX，而是产品模型本身存在偏差：

- 入口是阅读场景，运行模型却更像“先选模式再聊天”。
- 用户认为自己在自然提问，系统实际在隐式拼接多层上下文。
- 线程、模式、上下文、卡片、引用、确认动作、工具轨迹同时挤进一个窄 panel。
- “讲解 / 拆句 / 词义 / 语法 / 练习”被做成前置模式，而不是自然语言驱动下的能力调用。

因此 V2 的目标不是给现有 chatbox 换皮，而是重新定义 Ask Claread 在 Reader 里的产品角色。

## 当前实现诊断

以下判断基于当前代码实现。

### 1. 上下文模型过于隐式

当前 Ask 的上下文优先级实际为：

1. 用户显式附加的 `anchors`
2. `reader_focus`
3. 最近一条 assistant 消息中的 citation fallback

这意味着用户未必知道系统当前究竟基于哪句、哪段、哪条历史锚点在回答。

### 2. 模式前置导致聊天变硬

当前 Web 侧把 `讲解 / 拆句 / 词义 / 语法 / 练习` 作为发送前的显式选择层，导致：

- 用户在真正提问前被迫先做任务分类。
- 同一问题可能被模式错误约束。
- 模型缺少根据自然语言判断“该调用哪种教学能力”的空间。

### 3. 历史检索触发机制过脆

当前历史资产检索主要靠用户当前消息命中 regex 才放开，而不是一个稳定、可理解、可见的产品机制。

这会带来两个问题：

- 用户不知道怎样才能触发跨文章历史。
- 系统行为难以预测，也不利于后续升级为更稳健的检索策略。

### 4. 工具过程与真实过程不一致

当前前端展示的 tool trace 并不总能覆盖真实发生的查词、补卡和后处理过程，导致“为什么得到这个回答”不可解释。

### 5. 当前 chatbox 信息层级失控

同一个面板里同时承担：

- 线程切换
- 模式切换
- 对话
- 上下文摘要
- 来源
- 结构化卡片
- 保存确认动作
- 工具轨迹

这让 Ask 更像开发态运行时，而不是阅读态助手。

### 6. retry 语义不正确

当前“重新生成”本质上是重新发送上一条 user message，追加新回合，而不是对同一次 assistant 响应进行重试。

这会污染线程历史，也让用户对“重试”的预期落空。

## 对标结论

### Notion AI 可借鉴原则

- 一个 AI 心智模型，多入口复用。
- 默认吃环境上下文，用户再按需补充来源。
- 不强迫用户先选模式，模式是升级层。
- 回答与证据分层展示。
- 写操作先计划或确认，再执行。

### Langik 可借鉴原则

- `selection-first, action-second`
- 同一阅读线程可连续挂入多个选区
- 左侧原文，右侧助手，助手始终 book-bound
- 即时理解动作与后续练习动作分层

### Claread 不应照搬的部分

- 不应照搬 Notion 那种过宽的入口矩阵与控制面。
- 不应照搬 Langik 那种过长的动作菜单分叉。
- Claread 应比 Langik 更强调证据锚点、句段级引用和“本回答基于什么”的透明性。

## V2 产品定义

Ask Claread V2 是：

`解析页内、article-bound、自然语言驱动、可回源、可确认写入的阅读副驾驶。`

它不是：

- 泛聊天产品
- 独立 AI 页面
- 预设模式驱动的教学工具箱
- 默认全历史注入的学习总结器

它的核心价值是：

- 在用户读到阻塞点时，立刻围绕当前阅读环境给出解释。
- 在需要时，沿同一阅读线程继续追问，而不是跳出阅读场景。
- 在回答中保留足够的证据、锚点与可回跳能力。
- 在保存、摘录、收藏等写动作上维持明确确认。

## 核心原则

### 1. 自然语言优先

用户应该先提问，系统再决定：

- 当前问题更像解释、拆句、词义、语法还是练习
- 是否需要结构化卡片
- 是否需要查词典、本文资产、历史资产

而不是让用户先切一个模式。

### 2. 阅读环境优先

默认上下文不是“空 chat”，而是“当前阅读环境”。

优先级应为：

1. 当前显式选区或显式附加对象
2. 当前焦点句/焦点段
3. 当前章节或本文局部上下文
4. 仅在必要且可解释时扩展到历史资产

### 3. 上下文必须可见

系统可以自动推断上下文，但不能偷偷换上下文。

用户至少需要清楚知道：

- 本轮基于哪句或哪段
- 是否用了当前文章更多上下文
- 是否用了历史资产
- 是否用了词典或本文解析资产

### 4. 助手应贴着原文工作

桌面端 Ask 的主形态应是稳定的右侧阅读助手，而不是大多数情况下变成底部浮层。

### 5. 轻操作前置，重操作下沉

首层只保留阅读期最高频动作：

- 解释这句
- 这个词在这里什么意思
- 为什么这样写
- 继续追问

练习、保存、跨文章扩展、复杂检索与批量写入都应是第二层能力。

## 交互模型

### 入口

V2 保留两个主入口：

- 右侧 Ask 面板主入口
- 选区或句子上下文入口

选区入口的角色是“把当前阅读对象带入 Ask”，不是强行把用户送进某个固定模式。

### 空状态

空状态不应强调模式，而应强调当前阅读任务。

推荐展示：

- 一个自然语言输入框
- 2-4 个贴合阅读场景的 starter intents

例如：

- 解释这句在这里的意思
- 为什么作者这里这样写
- 这段和上一段是什么关系
- 围绕这一句出一道小练习

### 会话中状态

会话中只保留一个低干扰的上下文条，例如：

- 当前基于：这句 / 这段 / 本文
- 已附加：1 个选区
- 已扩展：历史资产

不再把“模式条”作为常驻主控件。

### 多锚点

同一线程允许继续追加新的句子、选区、卡片或资产对象。

但 UI 上应表达为“继续把这个也带进当前讨论”，而不是裸露 `anchor` 概念。

## 上下文模型

### 默认策略

默认只基于当前文章工作。

Ask 在一次回答中可读取：

- 当前选区
- 当前句与相邻句
- 当前段
- 本文章内相关解析卡、摘录、批注、收藏

### 历史扩展策略

跨文章历史不再只依赖 regex 命中。

V2 应采用：

- 自然语言意图识别
- 必要时显式上下文提示
- 可见的历史来源标记

系统行为应是：

- 默认不查历史
- 当用户明显在问“我之前见过吗 / 以前记过吗 / 这和我前面读的哪个地方类似吗”时扩展
- 一旦扩展，回答中要明确标记“使用了历史资产”

### 回退策略

如果当前轮没有显式选区，也没有明确焦点，系统可以使用最近讨论对象作为弱回退。

但必须满足两个条件：

- UI 可感知
- 若仍有歧义，优先澄清而不是强猜

## 能力模型

V2 不再把“讲解 / 拆句 / 词义 / 语法 / 练习”定义为首屏任务模式。

它们应被重定义为内部能力族：

- `explain_in_context`
- `breakdown_sentence`
- `explain_vocabulary_in_context`
- `explain_grammar_in_context`
- `generate_practice_from_context`

系统应根据用户自然语言、选区类型、上下文密度与回答目标，决定是否调用这些能力。

### 卡片策略

结构化卡片继续保留，但角色调整为“回答附属增强层”。

推荐保留：

- `sentence_breakdown_card`
- `vocabulary_in_context_card`
- `practice_card`

触发原则：

- 只有当问题确实需要这种结构时才渲染
- 卡片不替代正文回答
- 卡片默认弱于正文层级

## Target Harness Architecture

Ask Claread V2 的目标不是“一个更像 Notion 的聊天框”，而是一个贴在 Reader 内部的 assistant harness。

建议将运行架构明确拆成以下六层：

1. `surface layer`
2. `intent interpreter`
3. `context planner`
4. `capability orchestrator`
5. `retrieval layer`
6. `action / audit / eval layer`

### 1. Surface Layer

职责：

- 接收自然语言问题
- 展示当前工作上下文
- 展示回答、证据、确认动作
- 保持 article-bound 的阅读线程

这一层不负责：

- 让用户先选固定模式
- 暴露内部工具轨迹
- 决定底层检索策略

### 2. Intent Interpreter

职责：

- 根据用户自然语言判断本轮目标
- 判断是解释、拆句、词义、语法、比较、回顾、练习还是写动作
- 决定是否需要结构化卡片或确认动作

V2 的关键变化是：

- 前端不再以 `task_mode` 作为主入口
- 模式从用户显式选择项，降级为系统内部能力判定结果

### 3. Context Planner

职责：

- 构建本轮 working set
- 决定是否只看当前选区，还是扩到当前段、当前章节、本文资产
- 决定是否需要跨文章历史扩展
- 对最终采用的上下文给出可解释摘要

working set 的输出至少应包括：

- `primary_anchor`
- `local_context_window`
- `article_assets`
- `history_assets`
- `dictionary_context`
- `why_this_context`

### 4. Capability Orchestrator

职责：

- 统一调度阅读解释、词义、语法、练习、保存动作等能力
- 将工具调用、后处理卡片、确认动作收束到统一 contract
- 保证 tool result 和 UI result 不分裂

要求：

- 工具命名稳定
- 输入 schema 尽量窄
- 输出 shape 尽量确定
- 能力触发理由可记录、可评估

### 5. Retrieval Layer

职责：

- 为当前文章内问答提供确定性、高精度的局部检索
- 为跨文章历史问题提供可解释的候选召回和重排

这一层不等于“默认全局 RAG”。

它应先区分两类问题：

- 当前文章内 grounding
- 用户多解析页 / 跨文章历史 grounding

### 6. Action / Audit / Eval Layer

职责：

- 承载保存、收藏、摘录等写动作确认
- 记录 usage、billing、source labels、能力使用情况
- 为后续评测提供结构化日志

V2 需要在这一层新增：

- intent 判定质量评测
- context planning 质量评测
- retrieval 命中质量评测
- answer grounding 质量评测
- 用户阻塞是否被解决的反馈信号

## Retrieval / RAG Strategy

### 结论

Ask Claread 未来需要 retrieval layer，也大概率需要 RAG，但不是一开始就做“默认全历史向量化聊天”。

更合适的路线是：

- 当前文章内：以局部结构化上下文和确定性检索为主
- 跨文章历史：以 hybrid retrieval 为主，再逐步演进到 Claread-specific RAG

### 当前文章内

当前文章内问题通常不需要传统意义上的 RAG。

优先使用：

- 当前选区
- 相邻句 / 相邻段窗口
- 当前章节局部上下文
- 文章内解析卡、摘录、批注、收藏等结构化资产

原因：

- 精度高
- 可解释性强
- 证据更容易回跳
- 比通用 embedding 检索更符合 Reader 交互

### 跨文章历史

当用户问的是：

- 我之前见过这个表达吗
- 我之前记过这个词吗
- 这和我前面读过的哪里类似
- 帮我对比我之前在别的文章里的理解

这类问题需要真正的 retrieval pipeline。

建议分三阶段：

### Phase 1: Structured Retrieval

先做结构化和关键词检索：

- record / book / article metadata
- 章节和来源范围
- 资产类型过滤
- note / excerpt / favorite / vocabulary 标签
- 词形归一和精确匹配

### Phase 2: Hybrid Retrieval

在结构化检索基础上加入：

- lexical search
- embedding retrieval
- rerank

目标不是返回一堆命中，而是产出一个小而准的证据集。

### Phase 3: Claread-Specific RAG

当多解析页检索成为高频核心场景后，再进入真正的 RAG 组装：

- 召回候选
- rerank
- 组装 3-8 条最佳证据
- 生成带来源和命中理由的 context pack
- 把 context pack 交给回答层，而不是把全历史直接塞进 prompt

### RAG 约束

即使引入 RAG，也必须遵守以下约束：

- 默认不全量注入历史
- 历史扩展必须由意图驱动
- 回答必须标注历史来源
- 用户必须知道“用了哪些历史材料”
- 未命中时要明确说未命中，不能假装记得

## 当前架构差距

以下差距不是理想化要求，而是从当前实现直接推导出的重构目标。

### 1. 当前是 `task_mode-first`，不是 `intent-first`

当前请求 schema 仍以 `task_mode` 为主输入，前端也以模式条驱动发送。

结果：

- 用户被迫先分类
- 同一问题容易被错误模式约束
- 系统缺少“先理解问题，再决定能力”的空间

### 2. 当前是 fallback chain，不是 working set planner

当前上下文主要是：

- `anchors`
- `reader_focus`
- 最近 assistant citation fallback

结果：

- 用户难以知道 AI 当前到底在看什么
- 系统只是在补上下文，不是在规划上下文
- 不利于未来接入跨文章 retrieval

### 3. 当前 history lookup 是 regex gate，不是 retrieval policy

当前跨历史扩展主要由消息 regex 触发。

结果：

- 行为脆弱
- 命中范围不可控
- 用户无法建立稳定预期

### 4. 当前 capability contract 分裂

当前 tools 由 agent 侧硬编码，而部分 response cards 与 fallback proposals 又由 service 侧后处理补算。

结果：

- tool trace 和真实过程不完全一致
- UI 结果与内部运行步骤耦合混乱
- 后续新增能力时难以保持一致 contract

### 5. 当前持久化的是结果，不是决策过程

当前已经保存：

- `action_proposals`
- `tool_trace`
- `response_cards`
- `resolved_context`

但还没有保存：

- why this intent
- why this context
- why history was expanded
- why a capability was chosen

结果：

- 很难做高质量评测
- 很难定位回答失误是意图错、上下文错还是检索错

### 6. 当前 retry 语义与线程语义不干净

当前 regenerate 会追加新回合，而不是重跑原 assistant turn。

结果：

- 污染线程
- 放大历史噪声
- 不利于构建稳定 eval 样本

## 重构分期建议

### Phase A: Surface Cleanup

目标：

- 去掉模式前置
- 固定右 rail
- 上下文可见
- 证据层与正文层分开

这一阶段可以先复用现有大部分后端能力，但前端 contract 需要开始弱化 `task_mode`。

### Phase B: Intent + Context Planner

目标：

- 引入 intent-first 流程
- 将上下文选择从 fallback chain 升级为 working set planner
- 让历史扩展从 regex 机制升级为策略层机制

这是 Ask 从“chatbox”变成“assistant harness”的关键阶段。

### Phase C: Retrieval Upgrade

目标：

- 建立跨文章 hybrid retrieval
- 引入 evidence pack
- 为后续 Claread-specific RAG 打基础

### Phase D: Eval + Optimization

目标：

- 建立 Ask 专项评测集
- 评估 intent、context、retrieval、grounding、action proposal
- 再决定是否继续扩展工具和复杂能力

## 线程模型

### 默认线程

每篇文章保留一个主线程。

这个主线程代表“围绕当前文章的持续阅读对话”。

### New Chat

`New chat` 应保留，但不应作为首屏核心概念。

它适用于：

- 用户明确想换一个讨论方向
- 用户希望断开前文影响
- 用户开始一个新的长任务

### 历史与重试

`Retry / Regenerate` 应改为：

- 对同一 assistant turn 的重跑
- 不新增 user turn
- 不污染线程历史

## 回答与证据模型

### 回答主层

主层应始终优先输出：

- 直接答案
- 必要的简短解释
- 适合继续追问的收口

### 证据层

证据层默认折叠，但必须可用。

至少包括：

- 当前回答引用了哪句或哪段
- 是否用了本文资产
- 是否用了历史资产
- 是否用了词典

### 可回跳要求

只要回答引用了当前文章中的句子或段落，用户就应能回跳到相应位置。

### 工具过程展示

默认用户视图不应暴露完整 tool trace。

用户默认看到的应是“本回答使用了什么来源”，而不是“内部调用了几个工具步骤”。

完整 tool trace 只保留给：

- debug 态
- 内部联调
- 后续可能的开发者/高级模式

## 写动作模型

保留以下写动作：

- 保存为笔记
- 保存为高亮/摘录
- 收藏当前锚点
- 保存 assistant 回答为笔记

原则不变：

- 所有写动作都需要确认
- 确认卡应尽量低打断
- 确认目标必须清楚写明“保存到哪里、基于哪个锚点”

## 页面结构建议

桌面端建议采用稳定的三层关系：

1. 左/中：正文阅读区
2. 右：Ask Claread 阅读助手
3. 局部浮层：选区动作、轻释义、短暂提示

Ask 面板内部建议层级：

1. 顶部：标题 + 当前文章 + 少量线程控制
2. 中部：消息流
3. 底部：输入区 + 轻量上下文条

不建议在首层长期放置：

- 常驻模式条
- 长工具轨迹
- 多个大 disclosure 堆叠

## 明确不做

V2 当前不做：

- 独立 AI 工作台
- 默认全历史注入
- 复杂的手动工作集裁剪器
- 大而全的模式切换器
- 对话级人格化长期记忆
- 把练习系统和阅读助手彻底混成一个面板

## 评审关注点

后续重构方案评审时，应优先回答以下问题：

1. 用户是否可以不选模式，直接自然提问？
2. 用户是否能清楚知道本轮回答基于什么上下文？
3. Ask 是否真正是 article-bound 的阅读助手，而不是泛聊天框？
4. 卡片、来源、确认动作和工具信息是否层级清楚？
5. 线程与 retry 语义是否符合用户预期？
6. 历史扩展是否从“脆弱 regex”升级为“可解释的产品机制”？
